# Руководство по адаптации шаблона под варианты демоэкзамена

Этот файл нужен как рабочая инструкция. Его задача: быстро и без хаоса переделывать шаблон под новый вариант задания.

## Главное правило

Любое изменение в варианте нужно проводить по цепочке:

`задание -> Room Entity -> DAO -> репозиторий -> AppState -> UI -> проверка`

Если начать с UI, а структуру данных продумать потом, почти всегда придется переделывать форму и список второй раз.

## Порядок адаптации под новый вариант

### 1. Разобрать задание

Перед правками выпиши:

- какие сущности есть в варианте
- какие поля у каждой сущности
- какие связи между сущностями
- какие роли есть в системе
- кто что может просматривать
- кто что может добавлять, редактировать и удалять

Минимальная таблица для себя:

- сущность
- таблица
- экран
- роли доступа
- действия

### 2. Изменить структуру БД через Room

Используются файлы:

- `Entities.kt`
- `Relations.kt`
- `DatabaseConverters.kt`
- `AppDatabase.kt`

Как менять:

1. Если появилась новая сущность, добавь новую `@Entity`.
2. Если у сущности появилось новое поле, добавь его в соответствующую `@Entity`.
3. Если это справочник, делай отдельную таблицу.
4. Если есть связь, добавь `ForeignKey` и индекс.
5. Если нужен составной объект для экрана, опиши relation-класс в `Relations.kt`.

Что меняется чаще всего:

- `ProductEntity`
- `OrderEntity`
- `OrderItemEntity`
- `CustomerEntity`
- `PickupPointEntity`
- справочники категорий, поставщиков, производителей

### 3. Изменить DAO

Файл:

- `TemplateDaos.kt`

Как менять:

1. Добавь `@Insert`, `@Update`, `@Delete` для новой сущности.
2. Добавь только те `@Query`, которые реально нужны экрану.
3. Если нужна связанная выборка, используй `@Transaction` и relation-классы.
4. Не усложняй DAO лишними запросами.

Практический смысл:

- `Room` убирает ручной JDBC
- простые `@Query` для выборок остаются нормальной частью решения

### 4. Изменить репозитории

Чаще всего меняются:

- `AuthRepository.kt`
- `ProductRepository.kt`
- `OrderRepository.kt`
- `ReferenceRepository.kt`

Репозиторий должен:

- получить данные из DAO
- преобразовать `Entity` в модель экрана
- выполнить бизнес-правила
- отдать готовый результат в `AppState`

UI не должен работать с Room напрямую.

### 5. Изменить импорт из Excel

Файл:

- `XlsxImportMain.kt`

Когда менять:

- если изменился состав колонок
- если добавились новые справочники
- если поменялись названия листов
- если изменилась структура заказов

Как менять:

1. Открой пример `.xlsx` и выпиши точные названия столбцов.
2. Найди метод импорта для нужного файла.
3. Подстрой чтение ячеек под новый формат.
4. Если добавилось поле в сущность, обязательно добавь импорт этого поля.
5. Если поле справочное, сначала найди или создай запись справочника.

### 6. Изменить `AppState`

Файл:

- `AppState.kt`

Здесь нужно:

- хранить состояние экранов
- вызывать репозитории
- обновлять данные после CRUD-операций
- проверять права ролей
- управлять навигацией

Если добавляешь новый экран, почти всегда нужно менять `AppState`.

## Разработка UI

Это отдельный важный этап. Ниже порядок работы именно по интерфейсу.

### Где лежит UI

- тема: `src/main/kotlin/ru/demoexam/template/ui/theme/AppTheme.kt`
- общие компоненты: `src/main/kotlin/ru/demoexam/template/ui/components`
- экраны: `src/main/kotlin/ru/demoexam/template/ui/screens`
- навигация: `src/main/kotlin/ru/demoexam/template/navigation/AppScreen.kt`
- сборка интерфейса: `src/main/kotlin/ru/demoexam/template/app/App.kt`

### Что менять в UI в первую очередь

1. Заголовок приложения в `application.properties`.
2. Логотип и заглушку изображения в `src/main/resources/assets`.
3. Цвета и стиль в `AppTheme.kt`.
4. Структуру списка.
5. Структуру формы редактирования.
6. Подписи кнопок и сообщений.

### Какие экраны уже готовы

- `LoginScreen.kt`
- `ProductListScreen.kt`
- `ProductEditorScreen.kt`
- `OrderListScreen.kt`

### Как правильно дорабатывать экран

Экран должен:

- принимать данные параметрами
- отдавать действия через callbacks
- не обращаться к Room напрямую
- не хранить бизнес-логику доступа

Правильная схема:

1. Пользователь нажимает кнопку на экране.
2. Экран вызывает callback.
3. Callback идет в `AppState`.
4. `AppState` вызывает репозиторий.
5. Репозиторий работает с DAO.
6. `AppState` обновляет состояние.
7. Экран автоматически перерисовывается.

### Как добавить новый экран

1. Создай новый файл в `ui/screens`.
2. Добавь маршрут в `AppScreen.kt`.
3. Добавь состояние и обработчики в `AppState.kt`.
4. Подключи экран в `App.kt`.
5. Добавь кнопку перехода с существующего экрана.

### Что проверить по UI перед сдачей

- у каждого окна есть понятный заголовок
- с каждого экрана можно вернуться назад или на вход
- имя пользователя показывается в правом верхнем углу
- ошибки и предупреждения выводятся через диалог
- у ролей без прав скрыты или отключены запрещенные действия
- карточки и таблицы не ломаются на длинном тексте
- заглушка изображения отображается, если фото нет
- скидка и отсутствие товара подсвечиваются правильно

## Быстрые сценарии адаптации

### Если в варианте только товары

Меняй:

- `ProductEntity`
- DAO для товаров
- `ProductRepository.kt`
- `ProductListScreen.kt`
- `ProductEditorScreen.kt`

### Если нужен полный CRUD заказов

Меняй:

- `OrderEntity`
- `OrderItemEntity`
- DAO заказов
- `OrderRepository.kt`
- `OrderListScreen.kt`
- новый экран формы заказа

### Если главная сущность не товар

Например это клиенты, услуги, склады или запчасти.

Тогда:

1. Создай новую `Entity`.
2. Создай DAO под нее.
3. Сделай новый репозиторий.
4. Добавь экран списка.
5. Добавь экран формы.
6. Подключи их в навигацию.

## Мини-чеклист перед отладкой

1. `.\gradlew.bat compileKotlin`
2. `.\gradlew.bat test`
3. `.\gradlew.bat importXlsx -PimportDir="..."`, если нужен импорт
4. `.\gradlew.bat run`
5. Проверка входа под нужными ролями
6. Проверка поиска, фильтрации и сортировки
7. Проверка добавления, редактирования и удаления
8. Проверка сообщений об ошибках
